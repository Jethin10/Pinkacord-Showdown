# Pinkacord — Admin Panel Architecture

This document covers the Phase 3 admin panel that lives at
`tools/pinkacord-admin/`. The panel is a small web app that lets non-developers
manage all Pinkacord content (Pokemon, moves, abilities, items, formats) and
apply changes to a running PS server without ever touching code.

## Design pillars

1. **No new runtime dependencies.** The panel uses only Node's built-in
   `http` and `crypto` modules. This keeps merges from upstream PS clean and
   our security audit surface tiny.
2. **One source of truth.** The panel reads and writes the exact same
   `content/pinkacord/*.json` and `content/formats.json` files the server
   generator already consumes. It never holds a parallel database of mons.
3. **Schema-validated end-to-end.** Every API request body is validated with
   the same Zod schemas in `tools/pinkacord/schemas.ts`. An invalid edit is
   rejected with field-level errors before any file changes.
4. **Atomic builds.** "Save & Apply" triggers `tools/pinkacord/generator.ts`,
   which already writes atomically. If the generator fails the user sees the
   error and `content/` is unchanged.
5. **Auditable.** Every successful change appends to
   `logs/pinkacord/audit.jsonl` with timestamp, admin name, action, and a
   compact diff.
6. **Hotpatch coordination.** After a successful build the panel surfaces the
   exact `/hotpatch formats`, `/hotpatch battles`, `/hotpatch teamvalidator`
   commands an operator can paste into the PS chat. A future bot-account
   mode (Phase 3.5) will let the panel issue them itself.

## Authentication

v1 uses a **single shared admin password** read from `PINKACORD_ADMIN_PASSWORD`
environment variable. On startup the server hashes it with `crypto.scrypt` and
keeps the hash in memory. Logins compare via `crypto.timingSafeEqual`.

Session: server generates a 256-bit random session id, stores
`{ sid → { createdAt } }` in an in-memory `Map`, and sets a `pinkacord_session`
HttpOnly + SameSite=Strict cookie. Sessions expire after 12 hours of
inactivity. There's no "remember me" — admin sessions are short on purpose.

**Why so simple:** the panel runs alongside one PS server for one community.
A shared password is acceptable for v1; RBAC (per-admin accounts + roles) is
Phase 3.5 — see EDGE-CASES.pinkacord.md for the deferred list.


## API contract

All routes are JSON in / JSON out. Successful responses: `{ ok: true, ... }`.
Errors: `{ ok: false, code, message, fieldErrors? }`. Status codes follow
HTTP semantics (200/201/204/400/401/404/409/500).

Entity routes (one set per entity type — species, moves, abilities, items,
formats — generated by a shared factory):

| Method | Path                     | Body shape         | Notes                                |
|--------|--------------------------|--------------------|--------------------------------------|
| GET    | `/api/{entities}`        | —                  | list all                             |
| GET    | `/api/{entities}/:id`    | —                  | fetch one                            |
| POST   | `/api/{entities}`        | full entity        | create; 409 on id collision          |
| PUT    | `/api/{entities}/:id`    | full entity        | replace                              |
| DELETE | `/api/{entities}/:id`    | —                  | remove                               |

Build & hotpatch:

| Method | Path             | Effect                                                       |
|--------|------------------|--------------------------------------------------------------|
| POST   | `/api/build`     | Runs generator + PS rebuild + smoke test. Returns stats.     |
| GET    | `/api/hotpatch`  | Returns the exact `/hotpatch …` commands the operator should issue. |
| GET    | `/api/audit`     | Returns most recent N audit entries from JSONL log.          |
| GET    | `/api/effects`   | Lists registered effect kinds (used by ability/item forms).  |


## Audit log

`logs/pinkacord/audit.jsonl` is append-only. One JSON object per line:

```json
{ "ts": "2026-05-25T15:42:00.000Z", "actor": "admin", "action": "species.create", "id": "pinkachu", "before": null, "after": { ... } }
```

The Phase 3 panel's `/api/audit` endpoint tails this for the UI. Eventually
admin actions can be replayed from this log for incident response.

Backup advice: include `logs/pinkacord/` in any repo-backup workflow alongside
`content/`. It's the audit trail.


## Hotpatch coordination

PS has built-in hot reload commands (chat commands starting with `/hotpatch`)
that re-read data without dropping in-progress battles. After a successful
`/api/build` the panel displays:

```
✓ Build OK — to apply to the live server, paste:
    /hotpatch formats
    /hotpatch battles
    /hotpatch teamvalidator
```

This is v1. **Phase 3.5 will let the panel hold a bot-account WebSocket
connection** to PS and issue the commands itself.

If a hotpatch fails (e.g. because a battle holds references), PS responds in
chat — the operator decides whether to wait for battles to finish or do a
full server restart. We don't try to be cleverer than this for v1; the
content layer is already atomic so partial state is impossible.


## Concurrency model

- The admin panel runs as a single Node process. All write operations are
  serialized through the JS event loop.
- The content store uses **optimistic versioning**: every entity carries a
  `_rev` string (sha256 of its JSON when read). PUT/DELETE require matching
  `_rev` from the GET; if it has changed under you, the API returns 409 and
  the UI prompts to reload.
- The generator uses atomic temp-rename for all output files, so even a
  crash mid-build leaves the in-place files untouched.
- If two admins are editing the panel at once, both can browse simultaneously
  but a write that loses the `_rev` race must reload. This is enough for
  small teams (<10 admins). Hard locks (pessimistic) would be Phase 3.5.


## Security posture

| Concern                                        | Mitigation                                        |
|------------------------------------------------|---------------------------------------------------|
| Brute-force the admin password                 | scrypt hash with 64K cost; rate-limit failed logins (5 per 60s per IP) |
| XSS in admin-entered descriptions              | No HTML rendered from admin input — everything is `textContent` |
| Code injection via JSON                        | Zod schemas reject any non-allowlisted field; generator only emits known shapes |
| Session theft                                  | HttpOnly + SameSite=Strict cookie; 12h max; rotates on login |
| CSRF                                           | SameSite=Strict + custom header `X-Pinkacord-Admin: 1` required on mutating requests |
| Audit-log tampering                            | Append-only file; recommend log rotation + offsite backup |
| Path traversal in entity ids                   | IdSchema regex `[a-z0-9]+` blocks `..`, `/`, `\` |
| Disk-fill from huge JSON bodies                | Request body limit 1MB; longer descriptions truncate at schema's `.max()` |
| Admin runs panel on public internet            | Document: panel MUST be behind a firewall / VPN; not designed for public exposure |


## Deployment

- Configure with env vars (or a `.env` file the server reads):
  - `PINKACORD_ADMIN_PASSWORD` — required, the shared admin password
  - `PINKACORD_ADMIN_PORT` — default 8001
  - `PINKACORD_ADMIN_BIND` — default 127.0.0.1 (localhost only)
- Start with `npm run pinkacord-admin`.
- **Do not bind to 0.0.0.0 in production unless behind a VPN / firewall.**
- Recommended companion: a tunnel like Cloudflare Access or Tailscale to
  reach the admin panel remotely without exposing it publicly.


## What v1 covers

- Login screen with shared password
- Dashboard with summary counts
- Species, Moves, Abilities, Items, Formats — full CRUD
- Effect-kind builder UI for abilities (driven from `tools/pinkacord/effects.ts`)
- "Save & apply" button per entity (write JSON + rebuild + show hotpatch commands)
- Bulk "Build everything now" button
- Audit log viewer (last 100 entries)


## What v1 explicitly defers

- Per-admin accounts + roles → Phase 3.5
- Bot-account hotpatch automation → Phase 3.5
- Sprite upload UI → Phase 2.5
- Rich editor for description fields (markdown/preview) → Phase 4
- Multi-tenant (multiple communities sharing one panel) → out of scope
- Live preview of the mon in the teambuilder → out of scope
- Replaying audit entries to roll back → manual git revert for now
